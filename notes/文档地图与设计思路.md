# 文档地图与设计思路（通俗版）

## 1. 先把比喻说清楚：这个应用在「数据上」是什么形态？

你的需求里，**主版本**是：**数据主要在用户手机里**，不强制上云、也没有多端实时同步。  
AI 在文档里把这种形态叫做 **「本地优先 / 单机真源」**（local-first）：**课表、待办、日志等最终以手机里的存储为准**；服务器（如果有）更多是**帮忙**做教务登录、抓课表、OCR 识别这类「算完把结果给你」的事，而不是替你长期保管全部隐私数据——除非你以后明确做「云端真源」阶段。

理解这一点后，下面每份文档的角色就更好懂了。

---

## 2. 各文档分工一览（按「谁约束谁」排序）

### 2.1 `docs/prompts.md` — 变更日记 / 迭代流水账

- **在讲什么**：你和 AI 对话时，一条条编号记录「当时改了什么 UI/流程」。  
- **作用**：像**施工日志**；查「为什么日历上的今日改成描边了」这类历史原因时来这里。  
- **注意**：它不是「最终法律」；若和原型、需求文档打架，通常要以**原型 + 需求定稿**为准，再回头补 prompts。

### 2.2 `nju-timetable.pen`（原型文件）— 界面真源

- **在讲什么**：高保真界面长什么样、控件摆在哪。  
- **作用**：实现 UI 时**以它为准**（项目规则里也写了：界面真源是 pen）。  
- **和文档关系**：`requirements document.md` 里大量条目是在**把原型里的决策写成文字**，方便不写代码的人评审。

### 2.3 `docs/requirements document.md` — 产品需求说明书

- **在讲什么**：**做什么功能、不做什么**；删除入口在哪；哪些能力暂缓（例如课程社区在扩展 pen 里）。  
- **作用**：给产品/测试/你自己：**验收清单的源头**。  
- **特点**：已经把你早期想法里**砍掉**的东西写清楚了（云端备份、智能助手、「我的」主导航等）。

### 2.4 `docs/frontend-functional-spec.md` — 前端「行为规格」

- **在讲什么**：用户点哪里会发生什么（交互流）；路由级页面清单；主题/开关等**一致性约束**。  
- **作用**：开发时对照：**不是只画得像，还要行为对**。  
- **和需求文档关系**：需求说「要什么」，功能规格说「在界面上怎么流转」；二者应一致，细节以需求和原型为准。

### 2.5 `docs/data-and-state-contract.md` — 本地数据契约（偏存储）

- **在讲什么**：本地大概有哪些**存储键**、**实体最小字段**（课表、课程、待办……）；删改时哪些界面要一起刷新（invalidate）。  
- **作用**：防止「列表显示了 A、详情页还是旧的 B」这种数据不一致；也是和后端 DTO 对齐时的参照之一。

### 2.6 `docs/frontend-data-and-state-contract.md` — 前端状态契约（偏实现）

- **在讲什么**：在 `data-and-state-contract` 基础上更**贴近前端**：实体字段表、不变量（例如必须有一张主课表）、各页最小订阅状态。  
- **作用**：写 TypeScript 类型、状态管理、页面数据从哪读来对齐这份。

### 2.7 `docs/frontend-engineering-layout.md` — 工程目录与组件分层建议

- **在讲什么**：**不按具体框架写死**，但建议目录怎么分（`features/`、`domain/`、`data/`、`shared/`）；路由与原型画板编号怎么对应；性能注意点。  
- **作用**：多人协作或换框架时，仍能保持**依赖方向清晰**（例如：`domain` 不依赖 UI）。

### 2.8 `docs/backend-design.md` — 可选后端（分期）

- **在讲什么**：**阶段 A**：导入网关（教务、OCR）、节假日目录等，**一般不长期存用户课表**；**阶段 B**：若将来要上云同步再引入账户、同步协议等。  
- **作用**：你**还没写后端代码**时，先把边界、API 形状、安全原则想清楚；与前端 `CourseDraft` 等同构 DTO 减少胶水代码。

### 2.9 其他延伸文档（验收与结构）

- `docs/frontend-acceptance-matrix.md`：可勾选验收矩阵。  
- `docs/frontend-structure-and-components.md`、`docs/interaction-acceptance-checklist.md`：与工程布局/验收互补（需求文档里提到以最新定稿为准，避免重复维护）。

## 3. AI 是怎么「设计」这个应用的？（架构一句话 + 分层图）

### 3.1 一句话

**单机本地数据库（或等价持久化）为权威；界面按功能模块拆分；纯计算（周次、冲突分栏等）抽到领域层；需要学校系统或 OCR 时，用可选的「导入网关」把结果变成标准课程草稿，再由用户确认后写入本地。**

### 3.2 和 `frontend-functional-spec.md` 一致的四层（功能视角）

文档里写的是：

| 层级 | 含义（人话） |
| --- | --- |
| **展示层 View** | 页面、列表、按钮长什么样、怎么排版。 |
| **状态层 State** | 当前选哪张课表、第几周、主题色、弹窗开没开——**会变的内存状态**。 |
| **领域层 Domain** | **不依赖界面**的规则与计算：例如「这一周该不该显示这门课」「两门课时间重叠怎么分左右栏」。 |
| **数据层 Data** | **落盘**：写入 SQLite/文件、备份导出、以后可能有的网络请求适配器。 |

### 3.3 后端在图里的位置（可选）

`backend-design.md` 里的 Mermaid 图本质是：

**App → API 网关 → 几个服务（认证适配、教务导入、OCR、节假日目录）→ 校外系统（学校认证、教务网站）**

重点：**阶段 A** 强调服务端**尽量无状态、不把你的课表当主数据库**；和你产品「无云备份」是对齐的。

### 3.4 为什么这样拆？（原理，尽量短）

- **本地真源**：离线能用、隐私少上传、不依赖你养一个昂贵后台（MVP 友好）。  
- **Domain 独立**：换 UI 框架或改页面，**排课规则不用重写**。  
- **导入网关**：学校网站结构常变、登录复杂，集中在服务端做，比在每台手机里硬爬**好维护、好修**（也更涉及合规，见技术栈笔记里的「合规」一节）。  
- **DTO 与 `Course` 对齐**：减少「服务器一种字段名、客户端另一种」的 bug。

## 4. 你读文档时容易卡住的词（迷你词典）

- **真源（source of truth）**：以谁的数据为准。你项目当前是**手机本地**。  
- **DTO**：数据传输对象，API 往返时用的「结构化包裹」，字段名最好和客户端实体接近。  
- **BFF**：Backend for Frontend，给特定 App 量身定做的薄后端，聚合多个后端能力。这里「认证适配 + 导入」就类似 BFF。  
- **Invalidate / 刷新边界**：改了一处数据后，**哪些界面必须重新读数据**，契约里用表格约定了。  
- **CourseDraft**：导入得到的**草稿课**，用户确认合并前**不一定**等于最终写入的 `Course`。

## 5. 小结：读文档的「快捷键」

- 问「**做什么**」→ `requirements document.md` + 原型 pen。  
- 问「**点了会怎样**」→ `frontend-functional-spec.md`。  
- 问「**字段与存储键**」→ `data-and-state-contract.md` / `frontend-data-and-state-contract.md`。  
- 问「**文件夹怎么建**」→ `frontend-engineering-layout.md`。  
- 问「**服务器要不要、干什么**」→ `backend-design.md`。  
- 问「**为什么当时这么改**」→ `prompts.md`。
