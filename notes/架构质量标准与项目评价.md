# 架构质量标准与对本项目文档的评价

## 1. 「好架构」到底好在哪？——可操作的判断标准

专业里没有**唯一分数**，但业界常用下面几条衡量 **中小型应用架构** 是否健康。你可以把它们当成「体检表」。

### 1.1 是否对齐业务阶段（最重要）

- **好**：架构的复杂度**不大于**当前产品阶段需要的复杂度；预留扩展点但**不把未来当现在做**。  
- **差**：一上来就为「千万用户多区域多租户」搭架子，导致你三个月写不出 MVP。

**对你项目**：产品明确 **单机本地、无云备份**；架构若强行「云端真源」会**过度设计**。

---

### 1.2 边界是否清晰（高内聚、低耦合）

- **好**：UI、规则计算、存储、网络**谁改谁**一目了然；换 UI 不重写排课规则。  
- **差**：业务规则散落在按钮 `onClick` 里，改一个 bug 引出三个。

**对应文档**：`frontend-functional-spec.md` 的 View / State / Domain / Data 分层，`frontend-engineering-layout.md` 的依赖方向。

---

### 1.3 数据一致性与可恢复性

- **好**：关键操作**事务化**或等价保证；删改联动有**明确刷新规则**（invalidate）。  
- **差**：列表与详情各读各的缓存，用户看到「幽灵数据」。

**对应文档**：`data-and-state-contract.md` 第 1 节原则 + 第 4 节事件表。

---

### 1.4 安全与隐私「默认安全」

- **好**：敏感凭证短命、最小日志、TLS、限流；**假设网络与客户端都会被攻击**。  
- **差**：明文存教务密码、日志打印整张课表。

**对应文档**：`backend-design.md` §5、§8。

---

### 1.5 可测试性与可观测性

- **好**：`domain` 纯函数可单测；导入失败有可追踪 `requestId`；关键指标可监控。  
- **差**：全在组件里集成，无法单测；线上黑盒。

**对应文档**：`backend-design.md` §9；工程布局里建议 use-case 函数封装删除级联。

---

### 1.6 可演进性（升级成本可控）

- **好**：API `/v1`、DTO 向后兼容、`parserVersion` 区分教务解析版本；本地 `storageVersion` 迁移。  
- **差**：一改字段全端爆炸。

**对应文档**：`backend-design.md` §7；`data-and-state-contract.md` 持久化版本键。

---

### 1.7 文档能否驱动实现（可执行规格）

- **好**：新人能按文档搭出**一致**实现，争议少。  
- **差**：文档泛泛其词，开发各自发挥，最后与原型不一致。

**对应文档**：三份契约 + 功能规格 + 需求自查表——整体是**偏成熟**的拆分。

---

## 2. 对当前 AI 辅助产出架构的主观评价

### 2.1 总体结论

**对当前产品阶段（本地优先课表 App + 可选导入服务）而言，这套文档整体是「方向正确、边界清楚、可落地」的；属于良好的产品—工程衔接文档，而不是空中楼阁。**

它不是「性能最优」或「技术最潮」的架构，而是**风险与成本较平衡**的架构——这对个人或小团队通常是加分项。

---

### 2.2 做得好的地方（为什么算好）

1. **真源选择与产品一致**  
   明确单机存储为权威，后端阶段 A **不默认持久化课表**，避免和需求「无云备份」打架。

2. **分期后端（A 导入网关 / B 云同步）**  
   把「不得不碰学校系统」与「将来可能要账号体系」拆开，降低一次性投入与合规风险。

3. **DTO 与前端实体对齐**  
   `CourseDraft` 思路能显著减少集成 bug，这是做过集成的团队才会强调的。

4. **跨模块刷新表（invalidate）**  
   这是移动端复杂状态里**最容易翻车**的点，文档主动写出来非常实用。

5. **工程目录依赖方向**  
   `shared` 不引用 `features`、`domain` 不引用 UI——长期可维护性的基本功做对了。

6. **安全底线写进文档**  
   不存教务密码、短期会话、最小日志——说明作者有「默认不信任」意识。

---

### 2.3 薄弱点 / 需要你决策时补上的地方

1. **具体客户端栈未选定**  
   文档刻意中立；真正开工前你必须选 RN/Flutter/原生等——**选型不同，细节风险不同**（手势、通知、DB 插件成熟度）。

2. **教务导入的合规与稳定性是外部风险**  
   架构再漂亮也挡不住校方改版 HTML；需要 **parser 版本化**（文档已提）+ **用户可反馈/可重试**（功能规格也提了）。法律/ToS 要单独评估。

3. **「待办 ⟷ 考试作业」双写一致性**  
   契约里建议「源记录为真源，待办为投影」——**好方向**，但实现上要写清楚：删除源、编辑源、关闭开关时投影怎么更新，否则仍会产生幽灵待办。

4. **OCR 准确率与人工校对流程**  
   架构上异步 job 对了；产品上要强调 **预览确认**（后端文档也写了），否则用户会把错误课表写进真源。

5. **功能规格与需求偶有「实现细节级」表述需以原型为准**  
   例如某些筛选组合（如「当日」与其他筛选）要以**最新需求 + 原型**核对，避免开发误读。（这是**文档维护**问题，不是架构哲学问题。）

---

## 3. 给你的具体建议（按优先级）

### P0 — 开工前

- **选定客户端栈**（Flutter 或 RN 二选一对多数独立开发者更省）。  
- **选定本地 DB 方案**（SQLite 系最稳妥）并写好 **migration 策略**。  
- **把 domain 规则写成可单测的纯函数**（周次、冲突、showOnGrid 过滤）。

### P1 — 做导入前

- **理清教务导入合规**（学校 ToS、用户告知、数据最小化）。  
- 按 `backend-design.md` 实现 **ImportProvider 抽象**：`mock` → 真网关，避免 UI 被网络阻塞。

### P2 — 上线前

- **备份/导出**格式与恢复路径要实测（需求里有本地备份）。  
- **删除级联**按需求第十四章逐条用例测（文档已列表，很适合当测试清单）。

### P3 — 若做云同步（未来）

- **不要**在没想好冲突策略前上云；阶段 B 单独立项，文档已提示——听从该提示。

---

## 4. 一句话记住「好架构」

> **好架构 = 在正确的产品阶段，用清晰的边界把数据与安全管住，并让明天的改动主要发生在一个地方。**

你当前仓库里的文档，在这条线上的得分是**偏高的**；真正的考验会在 **具体栈选型后的实现细节** 与 **教务/OCR 外部依赖** 上。
