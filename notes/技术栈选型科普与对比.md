# 技术栈选型科普与对比（详尽笔记）

> 说明：仓库里的工程文档**故意没有锁死**具体框架版本，以便你决策。下面按「你这个课表 App 的典型需求」组织：**移动端优先、本地持久化、复杂列表与日历、可选服务端导入**。

---

## A. 客户端形态：原生 / 跨平台 / Web 壳

### A.1 纯原生（Swift + Kotlin 各写一套）

**是什么**：iOS 用 Swift/SwiftUI，Android 用 Kotlin/Jetpack Compose，两套代码库。

| 优点 | 缺点 |
| --- | --- |
| 性能、系统 API（通知、小组件、后台任务）最顺 | 两套团队/两套 bug；功能迭代成本几乎翻倍 |
| 平台 UI 规范最「正宗」 | 业务逻辑重复，需自律抽「共享内核」（少见且难） |

**业界何时选**：大厂核心 App、对端上体验极致、预算足。

**对你项目**：功能面多（课表网格、拖拽、本地 DB、提醒）但**不是游戏级性能**；除非你就想练两门原生，否则**性价比偏低**。

---

### A.2 Flutter（Dart）

**是什么**：自己画 UI，编译到 iOS/Android（及更多平台）；通过 **platform channel** 调系统能力。

| 优点 | 缺点 |
| --- | --- |
| 一套代码双端，UI 一致性极好 | Dart 生态小于 JS；部分原生新特性要等插件 |
| 动画与自定义布局强，画课表网格友好 | 包体体积通常比纯原生大 |
| 性能整体接近原生（Skia 渲染） | 与系统「完全原生质感」仍有细微差别（见仁见智） |

**原理简述**：不是把 WebView 包起来，而是**运行时 + 渲染引擎**画像素，所以比「纯 H5 壳」流畅。

**对你项目**：课表、日历、主题令牌、复杂手势（长按拖拽）都合适；本地存储有成熟方案（sqflite、drift、isar 等）。

---

### A.3 React Native（JS/TS + Hermes）

**是什么**：用 React 写界面，桥接到原生组件；新架构（Fabric/TurboModules）缩小与原生差距。

| 优点 | 缺点 |
| --- | --- |
| 若团队会 React/TS，上手快 | 复杂手势、超重度列表有时要原生补丁或调优 |
| 生态大（npm），招外包/协作容易 | 升级 RN 大版本有历史债风险 |
| 与 Web 共享部分逻辑（有限） | 「一次编写到处运行」仍要处理平台差异 |

**对你项目**：完全可行；课表需注意**列表/网格性能**与**重渲染**（用 memo、selector、虚拟化列表等）。

---

### A.4 Kotlin Multiplatform Mobile（KMP）+ 各端 UI

**是什么**：业务逻辑、网络、存储模型用 Kotlin 共享；UI 仍可用 Compose（Android）与 SwiftUI（iOS）。

| 优点 | 缺点 |
| --- | --- |
| 逻辑真·共享，类型安全 | 总复杂度高于「单栈跨平台」；团队要会 Kotlin + 双端 UI |
| 适合「规则多、UI 也要原生」 | 学习曲线陡，小团队启动慢 |

**对你项目**：若核心在 **domain 规则**（周次、冲突、同步），且你坚持**原生 UI**，KMP 是中长期选项；MVP 偏重。

---

### A.5 Web 技术 + 壳（Capacitor / Ionic）

**是什么**：主要用 HTML/CSS/JS 开发，壳进 App Store；通过插件访问相机、文件等。

| 优点 | 缺点 |
| --- | --- |
| Web 栈统一，甚至可上架简单 Web 版 | 复杂滚动、手势、高性能动画不如 Flutter/RN 稳 |
| 迭代快 | 课表密集网格 + 拖拽要格外小心性能 |

**对你项目**：能做，但**课表 + 拖拽 + 离线 DB** 组合下，工程上往往比 Flutter/RN **更费调优**。

---

### A.6 小结（客户端建议方向）

- **想最快双端 + 自定义 UI 强**：Flutter 或 React Native。  
- **想极致系统整合 + 不差人**：双原生或 KMP。  
- **想优先 Web 再打包**：Capacitor，但要接受性能调优成本。

---

## B. 本地持久化：键值 / SQL / NoSQL

你的契约文档里有多实体、关联、**事务化写入**要求——这不是「只存一个 JSON」能轻松搞定的。

### B.1 轻量键值（SharedPreferences / UserDefaults / MMKV）

**适合**：主题色、开关、当前周索引等小配置。  
**不适合**：整份课表、待办、日志（易丢一致性、难查询）。

---

### B.2 嵌入式 SQL（SQLite）

**是什么**：手机里的迷你关系型数据库；支持事务、索引、查询。

| 优点 | 缺点 |
| --- | --- |
| 模型清晰，关联查询自然 | 要写 migration（`storageVersion` 升级脚本） |
| 与「课表—课程—待办关联」契合 | ORM 选不好会增加样板代码 |

**常见组合**：

- **React Native**：`react-native-quick-sqlite`、`expo-sqlite`、Drizzle ORM 等。  
- **Flutter**：`sqflite`、`drift`（类型安全查询）。  

**原理**：ACID 事务保证「删课程 + 清关联待办」要么全成要么全败，避免半写入。

---

### B.3 对象型 NoSQL（Realm、ObjectBox、Isar）

**是什么**：以对象为单位存储，查询 API 各不同。

| 优点 | 缺点 |
| --- | --- |
| 开发体验有时比 SQL 快 | 复杂关联与报表式查询可能别扭 |
| 同步功能（若用厂商方案）可省部分自建 | 版本迁移、备份格式要想清楚 |

---

### B.4 WatermelonDB（偏 RN）

**是什么**：为 React Native 优化的懒加载 DB，适合超长列表。

| 优点 | 缺点 |
| --- | --- |
| 列表性能口碑好 | 学习曲线；模型定义方式要适应 |

**对你项目**：若待办/日志列表极长且要极致流畅，可评估；一般 SQLite + 良好分页也够用。

---

## C. 状态管理（前端内存中的「当前世界」）

### C.1 要解决的问题

- 多页面共享：**当前课表 id、第几周、主题模式**。  
- 一致性：**删课程**后课表页与「全部课程」页都要更新。  
- 性能：课表网格避免「改一个开关重算整个学期」。

### C.2 常见选项

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **React Context + useReducer** | 无依赖、简单 | 规模大时 Provider 地狱、重渲染难控 |
| **Redux / Redux Toolkit** | 可预测、时间旅行调试、生态成熟 | 样板多（RTK 已缓解） |
| **Zustand / Jotai / Recoil** | 轻、上手快 | 超大团队需自律约定模式 |
| **MobX** | 响应式直观 | 魔法感强，调试路径与 Redux 不同 |
| **Riverpod / BLoC（Flutter）** | 结构清晰、测试友好 | 学习成本 |

**与文档对齐**：`frontend-engineering-layout.md` 里建议 **按 feature 切片** 或单一 store——**Redux Toolkit** 与 **Zustand** 都容易做到；关键是 **selector / memo**，不是哪个库名字。

## D. 导航与路由

- **React Native**：React Navigation（主流）、Expo Router（文件路由）。  
- **Flutter**：GoRouter、Navigator 2.0。  

**对你项目**：二级页多、返回栈要明确；需求里「重进 App 回默认本人课表」＝要注意**冷启动恢复状态**策略，而不是只依赖内存路由栈。

---

## E. 通知与后台（上课提醒）

**原理**：系统级 **本地通知**（Local Notifications）由 OS 调度；精确触发依赖各系统省电策略与权限。

| 点 | 说明 |
| --- | --- |
| **可靠性** | iOS/Android 对后台限制不同；需查当前系统「精确闹钟」权限（Android 13+ 等） |
| **实现** | 通常用成熟插件（如 RN 的 notifee、Flutter 的 flutter_local_notifications） |
| **与数据** | 提醒应基于本地课表计算「下一次事件」，并在课表变更时**重调度** |

## F. 可选后端：语言与形态

文档 `backend-design.md` 描述的是 **API 网关 + 无状态导入服务**。实现语言常见选择：

### F.1 Node.js（TypeScript）

| 优点 | 缺点 |
| --- | --- |
| 与前端同语言，DTO/Zod 可共享 | CPU 密集任务（大批量 OCR）要放队列或转其他语言 |
| 生态极大 | 运行时单线程，要注意阻塞 |

**适合**：BFF、聚合、JWT、与前端字段高度一致。

---

### F.2 Go

| 优点 | 缺点 |
| --- | --- |
| 部署简单、并发好、性能稳 | 类型模型与前端共享需 code gen 或手写 |
| 适合网关、worker | 业务快速迭代有时不如 Node「快写」 |

**适合**：API 网关、导入 worker、高并发限流。

---

### F.3 Python（FastAPI 等）

| 优点 | 缺点 |
| --- | --- |
| 写爬虫/解析原型快 | 生产部署与并发模型要用心（仍可做，很多团队用） |
| AI/OCR 流水线生态好 | 类型与前端对齐靠 discipline |

**适合**：**OCR、HTML 解析实验**、内部工具；也可生产，但需规范。

---

### F.4 「无服务器」vs 自建 VPS

| 形态 | 优点 | 缺点 |
| --- | --- | --- |
| **Serverless（云函数）** | 按量计费、免运维 | 冷启动、长连接、大文件上传要设计 |
| **容器（Docker + K8s/单节点）** | 可控、易迁移 | 运维成本 |
| **单机 VPS** | 简单 | 扩展性、可用性要自己做 |

## G. 教务导入与合规（比技术更重要的「原因」）

**原理**：学校教务系统通常有 **服务条款** 与 **反爬虫策略**；统一认证（CAS/OAuth）也有安全要求。

| 做法 | 说明 |
| --- | --- |
| **官方开放 API** | 最理想，少见 |
| **校方合作** | 最正规 |
| **用户授权下的服务端代理** | 工程上常见，但仍需法律/合规评估 |
| **客户端内嵌 WebView 让用户自己登录** | 减少你碰密码的概率，但不自动消灭 ToS 问题 |

文档强调 **不在自家 DB 存教务密码**、**短期 token**、**最小日志**——这是**安全与合规上的正确方向**。  
**建议**：产品对外说明「导入经服务器中转」、数据留存周期、是否记录学号等（`backend-design.md` 已提到隐私政策需求）。

## H. OCR 路径

| 方式 | 优点 | 缺点 |
| --- | --- | --- |
| **设备端 ML（离线）** | 隐私好、无服务器成本 | 模型体积、准确率难控 |
| **云端 OCR API** | 准确率高 | 费用、隐私、网络依赖 |
| **自建异步任务** | 可控 | 运维与队列成本 |

文档建议 **异步 job + 轮询/回调**，符合 OCR 耗时特性。

## J. 若未来做「云同步」（阶段 B 预告）

业界常见同步策略：

| 策略 | 原理 | 代价 |
| --- | --- | --- |
| **Last-write-wins** | 谁最后改算谁的 | 易丢编辑，需产品接受 |
| **向量时钟 / 版本号** | 每条记录 `updatedAt` + 墓碑删除 | 需冲突合并 UI 或规则 |
| **CRDT** | 数据结构数学上可合并 | 学习成本高，未必适合所有字段 |
| **端到端加密** | 密钥只在客户端 | 服务端看不到明文，找回密码难 |

你的 `backend-design.md` 已点名阶段 B 要单独立项评审——这是**负责任**的。
